================================================================================
🩺 DermaScreen — System Architecture & Workflow Specification
================================================================================

This document provides a comprehensive technical overview of the DermaScreen 
system, including its high-level architecture, component relationships, data 
pipelines, authentication mechanisms, and external service integrations.

--------------------------------------------------------------------------------
1. SYSTEM ARCHITECTURE
--------------------------------------------------------------------------------

DermaScreen is structured as a decoupled client-server application powered by 
a React frontend, a FastAPI backend, and cloud-hosted data/auth services.

2. KEY COMPONENTS
--------------------------------------------------------------------------------

A. Frontend Application (React 19)
----------------------------------
* App Core (App.jsx / main.jsx): Configures application routes using React Router 
  DOM v7 and wraps the interface in the AppContext provider.
* State Management (AppContext.jsx): Manages session state, logged-in user profile, 
  active scans, dashboard statistics, and real-time chat history.
* Firebase Authentication Client (services/firebase.js): Manages sign-up, sign-in 
  (email/password and Google OAuth), password resets, and retrieves JSON Web 
  Tokens (JWT) for secure backend requests.
* API Client (services/api.js): Employs an HTTP client to communicate with the 
  FastAPI backend, automatically injecting the user's Firebase ID Token into the 
  "Authorization: Bearer <token>" header of every request.

B. Backend Application (FastAPI)
---------------------------------
* Entry Point (app/main.py): Runs on Uvicorn, configures CORS middleware to allow 
  cross-origin requests from defined frontend URLs, and exposes application routes.
* Lifespan Events (app/main.py): Preloads core resources on startup to minimize 
  request latency:
    1. Initializes the Firebase Admin SDK using service account credentials.
    2. Loads image quality config/thresholds (image_quality.pkl).
    3. Preloads the PyTorch ResNet50 classifier weights (model.pth) and model 
       configuration parameters (grad_cam.pkl) into memory.
* Token Verification Dependency (app/dependencies.py): Intercepts request headers, 
  decodes and validates Firebase JWT tokens, extracts the user ID/email, and 
  registers or retrieves the corresponding user record from Supabase.
* Service Layer (app/services/):
    * quality_service.py: Analyzes uploaded image files using OpenCV.
    * ml_service.py: Preprocesses images, runs PyTorch classification, integrates 
      the clinical ABCDE rule, and generates Grad-CAM visualizations.
    * supabase_service.py: Manages connections to Supabase PostgreSQL database 
      and Storage buckets.
    * chat_service.py: Sends structured conversational payloads (including text, 
      history, and image assets) to OpenRouter.

--------------------------------------------------------------------------------
3. CORE PIPELINES & DATA FLOWS
--------------------------------------------------------------------------------

A. Skin Lesion Upload & AI Analysis Pipeline
---------------------------------------------
The primary workflow processes user-uploaded skin scans through validation, 
storage, deep learning prediction, and database logging:

1. User Action: The user selects a lesion image, answers the optional 5-point 
   clinical ABCDE questionnaire, and clicks "Analyze".
2. Client Request: The React UI wraps the image file and ABCDE answers in a 
   multipart/form-data request, attaches the Firebase ID token in the 
   "Authorization: Bearer <Token>" header, and sends it to /api/v1/predict/.
3. Auth Verification: FastAPI intercepts the request, verifies the token via the 
   Firebase Admin SDK, and fetches/upserts the user record in Supabase.
4. Image Quality Assessment (OpenCV):
   * Resolution Check: Rejects/warns if resolution is below 224x224px.
   * Blur Detection: Calculates Laplacian variance (calibrated via 
     image_quality.pkl) to identify blurry images.
   * Lighting Quality: Calculates mean brightness to flag under- or over-exposed 
     images.
   * Lesion Visibility: Applies Gaussian blur + Canny edge detection + contrast 
     checks to verify if a visible lesion structure is present.
5. Storage Persistence (Supabase Storage): The original image bytes are uploaded 
   to the private Supabase Storage bucket, returning a public image URL.
6. ML Inference & ABCDE Fusion:
   * The image is normalized and passed through the ResNet50 classifier to 
     predict the suspicion probability.
   * The suspicious probability is adjusted based on the clinical ABCDE score:
     - Score = 0: Suspicion probability reduced by 75%.
     - Score = 1: Suspicion probability reduced by 50%.
     - Score = 2: No modification.
     - Score = 3: Minimum suspicion probability set to 60%.
     - Score = 4: Minimum suspicion probability set to 80% (High Risk).
     - Score = 5: Forced to 95% (Urgent Evaluation).
   * Map the adjusted probability to a Risk Tier (Low, Moderate, High, Urgent).
7. Grad-CAM Visualization: The model computes gradients of the target class 
   with respect to the final convolutional layer feature maps, blends the jet 
   color-mapped activations 40/60 onto the original image, and returns it as a 
   Base64-encoded PNG image.
8. DB Log: The scan record metadata (Scan ID, Prediction, Risk, Image URL, 
   Quality metrics) is saved to the Supabase database.
9. Response: The API returns the completed PredictResponse schema, and the UI 
   displays the interactive result dashboard.

B. Multimodal AI Chat & Context Grounding Pipeline
---------------------------------------------------
1. User Action: The user submits a question or uploads an image in the AI Chat.
2. Client Request: The UI retrieves the user's scan history and sends the chat 
   message, history, and optional attachment url to /api/v1/chat/.
3. Context Construction: The backend gathers the patient's latest scan metadata 
   from Supabase and formats it into the system prompt to ground the LLM.
4. Multimodal LLM Request: The chat service builds a payload containing the 
   grounding system instructions, conversation logs (last 10 turns), and the 
   new multimodal message, and sends it to OpenRouter (Gemini 2.0 Flash).
5. Output & Safety Disclaimer: The model replies with empathetic medical 
   information, suggestions for over-the-counter options/remedies for common 
   issues, and structures the response. A disclaimer recommending clinical 
   evaluation is appended before sending the reply to the frontend.

--------------------------------------------------------------------------------
4. SECURITY & AUTHENTICATION ARCHITECTURE
--------------------------------------------------------------------------------

* Token-Based Authorization: The application uses a zero-trust architecture. The 
  frontend client authenticates with Firebase to retrieve a JWT, which is 
  included in all secure requests.
* Server-Side Validation: FastAPI verifies all tokens with the Firebase Admin 
  SDK's verification logic.
* Database Tenant Isolation: Every query to Supabase database/storage filters 
  results by the verified user's ID to prevent cross-user data leakage.
